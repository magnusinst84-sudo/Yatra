# Yatra Architecture Overview

Yatra is a full-stack AI walkthrough application that generates historically grounded, location-based interactive journeys. It is built using a modern React frontend and a FastAPI (Python) backend, supported by MongoDB for caching and Firebase for authentication.

## 1. High-Level Flow (The User Journey)
1. **Search:** A user selects a Place and Era on the frontend (e.g., "Kyoto" in "Medieval").
2. **Global Cache Check:** The backend checks MongoDB to see if this exact journey has already been generated by another user (where `user_uid="system"`). If so, it is served instantly (0s wait).
3. **Retrieval-Augmented Generation (RAG):** If no cache exists, the backend queries a local Chroma vector database to fetch historically accurate context about that specific place and time.
4. **Agent Generation:** A Gemini AI agent is prompted with the RAG context to generate a structured 3-to-5 stop walkthrough in valid JSON, complete with second-person narration, location-specific daily life facts, and scene-setting image prompts.
5. **Image Generation:** The image prompts are passed asynchronously through a resilient fallback chain (Imagen 4 -> Pollinations.ai -> Transparent Placeholder).
6. **Persistence:** The final JSON state, including the Base64 image URLs, is saved into MongoDB's global pool (`user_uid="system"`) and returned to the frontend.

## 2. Component Architecture

### Frontend (React + Vite + Tailwind)
The frontend uses state-based conditional routing (no heavy routing libraries) to maintain a fast, single-page application feel.
* **Authentication:** Handled entirely by Firebase (Google OAuth). The JWT `idToken` is sent in the `Authorization` header to protected backend endpoints.
* **State Management:** `App.tsx` manages the global `user` and `idToken` context, rendering the `<Dashboard>` or `<LandingPage>` accordingly.
* **Modal Overlay:** The immersive walkthrough is rendered in a full-screen `<WalkthroughModal>` overlay that is injected at the top level, allowing guest users clicking on shared links to view experiences over the landing page without authenticating.

### Backend (FastAPI + Pydantic)
* **API Layer (`backend/api/walkthrough.py`):** A RESTful API that handles fetching supported places, starting generations, claiming ownership (saving), and generating share slugs.
* **RAG Pipeline (`backend/rag/`):** Manages vector embeddings and context retrieval. Runs synchronously in a thread pool to avoid blocking the `asyncio` event loop.
* **Agent Engine (`backend/agent/`):** Connects to the `google.genai` SDK. Employs a strict validation loop ensuring Gemini returns the exact requested JSON schema with historical continuity between stops.
* **Image Fallback Pipeline (`backend/services/image_gen.py`):** Maximizes visual generation success by cascading down from premium models (Imagen) to free-tier HTTP APIs (Pollinations), and finally to a static placeholder to prevent frontend rendering crashes.

## 3. Data Persistence (MongoDB)
* **Global Scope:** Newly generated walkthroughs are owned by `system`. These act as a global cache.
* **User History (Cloning):** When a user clicks "Save" on a walkthrough, the backend clones the `system` document, assigns a new UUID, and sets the `user_uid` to the user's Firebase UID. This populates their "History" tab without stealing the original from the public cache.
* **Share Links:** Generating a share link creates an 8-character string (slug) attached to the walkthrough document, allowing unauthenticated `GET` requests to retrieve it via `/api/walkthrough/shared/{slug}`.
